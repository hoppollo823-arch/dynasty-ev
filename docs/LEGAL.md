## 1. 地图数据授权

### 1.1 OpenStreetMap — ODbL 合规

**数据性质：** OSM 社区数据，采用 **开放数据库许可证 (Open Database License, ODbL 1.0)**。

**ODbL 核心要求：**

| 要求 | 说明 | DynastyVR 合规措施 |
|------|------|--------------------|
| **署名 (Attribution)** | 必须标注数据来源 | 在游戏设置/关于页面标注 "© OpenStreetMap 贡献者" |
| **共享相同方式 (ShareAlike)** | 衍生数据库须以相同许可发布 | 地图处理管线的中间数据集须开源或不对外发布 |
| **开放 (Open)** | 修改后须公开修改 | 若对外发布修改后的 OSM 数据，须以 ODbL 发布 |

**合规清单：**

- [ ] 在游戏加载画面/关于页面添加 OSM 署名
- [ ] 确认 OSM 数据仅用于内部分析/渲染，不直接再分发原始数据库（若再分发须 ODbL）
- [ ] Cesium for Unreal 内置 OSM 署名处理 — 验证其是否自动包含归属信息
- [ ] 若使用 OSM Building 数据（3D 建筑），同样需 ODbL 合规
- [ ] 定期同步 OSM 数据，保持与上游变更一致

**风险等级：** 🟡 中 — ODbL ShareAlike 条款需注意衍生作品的许可证

### 1.2 Sentinel-2 卫星影像 — 开源许可

**数据性质：** 欧空局 Copernicus 计划，Sentinel-2 数据采用 **Creative Commons Attribution 4.0 International (CC BY 4.0)**。

**CC BY 4.0 要求：**

- **署名：** 须标注 "Contains modified Copernicus Sentinel data [年份]"
- **商用：** ✅ 允许商用
- **修改：** ✅ 允许修改和衍生作品
- **无 ShareAlike：** ✅ 不要求衍生作品使用相同许可

**合规措施：**

- [ ] 在游戏"关于数据"页面添加 Copernicus Sentinel 署名
- [ ] 影像处理流水线保留原始元数据（产品 ID、获取时间）以备溯源
- [ ] 若使用 Google/Bing Maps 底图影像作为补充，需另查其 API 许可（见 1.3）

**风险等级：** 🟢 低 — CC BY 4.0 宽松友好

### 1.3 Google Maps / Bing Maps API

**Google Maps Platform：**

| 要求 | 说明 |
|------|------|
| **API 密钥** | 必须使用有效 API 密钥，禁止绕过 |
| **署名** | 必须在显著位置显示 Google 地图 Logo 及 "Google" 字样 |
| **禁止转售** | 不得将原始地图瓦片作为独立产品转售 |
| **3D Tiles** | Google Photorealistic 3D Tiles（通过 Cesium）需额外确认许可条款 |
| **付费** | 按加载量计费；需预算控制 |
| **SLA** | 须遵守 Google Maps ToS 的可用性条款 |

**Bing Maps API (Microsoft)：**

| 要求 | 说明 |
|------|------|
| **API 密钥** | 同上 |
| **署名** | 须显示 "© Microsoft" |
| **使用限制** | 不得用于实时导航（DynastyVR 场景可豁免） |
| **3D Tiles** | Bing Maps 的 3D 建筑数据可通过 Cesium 集成使用 |

**DynastyVR 合规策略：**

1. **优先使用 Cesium for Unreal 内置资源** — Cesium 已与 Google/Bing 签署分发协议，通过 Cesium 的 Ion Token 使用 3D Tiles 可自动合规
2. **游戏内 UI 避免遮挡地图 Logo** — 如使用 Google Tiles，需在 UI 设计中预留 Logo 区域
3. **设立 API 调用预算上限** — 防止失控费用
4. **备选方案评估**：考虑使用 Mapbox GL 或 OpenStreetMap 自建瓦片服务以降低授权成本

**风险等级：** 🔴 高 — 商业 API 条款严格，违规可能导致服务中断或法律追诉

## 2. 天体数据使用

### 2.1 NASA / ESA 数据开放政策

**NASA 数据政策：**

| 数据集 | 许可 | 商用 | 署名要求 |
|--------|------|------|----------|
| LRO LOLA 月球地形 | 公共领域 | ✅ | 建议署名（非强制） |
| MOLA 火星地形 | 公共领域 | ✅ | 建议署名（非强制） |
| 星表 (Hipparcos/Tycho-2) | 公共领域 | ✅ | 无 |
| ERA5 再分析数据 (ECMWF) | CC BY 4.0 | ✅ | 须署名 |
| MODIS 植被数据 | NASA Open Data | ✅ | 建议署名 |

**NASA 关键条款：**

- NASA 图像/数据/音频/视频一般为 **公共领域**，无版权限制
- **例外：** NASA Logo（肉丸 Logo、蠕虫 Logo）**受法律保护**，不得未经许可使用
- 不得暗示 NASA 为项目背书或赞助
- NASA 名称可用于事实性描述（如"使用 NASA LRO 数据"）

**ESA / Copernicus 数据：**

| 数据集 | 许可 | 署名 |
|--------|------|------|
| Copernicus DEM (30m) | CC BY 4.0 | 须署名 |
| Sentinel-2 影像 | CC BY 4.0 | 须署名 |

**合规清单：**

- [ ] 在天文模块"数据来源"页面标注各数据集来源
- [ ] 不使用 NASA Logo 作为项目品牌元素
- [ ] 在宣传材料中避免"NASA 官方合作"等误导性表述
- [ ] ERA5 数据标注 "ECMWF ERA5 Reanalysis"

**风险等级：** 🟢 低 — 大部分为公共领域，合规简单

### 2.2 EHT（事件视界望远镜）影像版权

**EHT 黑洞影像：**

- **首张黑洞影像（M87\*，2019）：** EHT 合作组发布，采用 **CC BY 4.0** 许可
- **银河系中心黑洞（Sgr A\*，2022）：** 同样 CC BY 4.0
- **署名要求：** "Event Horizon Telescope Collaboration" + 具体论文引用

**使用注意事项：**

1. **科学准确性：** EHT 影像是真实观测数据的可视化呈现，DynastyVR 中使用时应避免过度艺术化修改导致科学误导
2. **衍生可视化：** 如基于 EHT 数据创建引力透镜动画，须注明"基于 EHT 观测数据的科学可视化"而非"真实黑洞照片"
3. **学术引用：** 建议在技术文档中引用原始论文（EHT Collaboration et al., ApJL, 2019）
4. **商业用途：** CC BY 4.0 允许商用，但须署名

**合规措施：**

- [ ] 黑洞场景加载时显示 EHT 署名浮层
- [ ] 科学顾问审核黑洞可视化内容的准确性
- [ ] 不在未经 EHT 合作组许可的情况下使用其 Logo

**风险等级：** 🟡 中 — 需确保衍生可视化不构成科学误导

## 3. UE5 许可

### 3.1 Epic Games EULA 核心要求

**许可类型：** Unreal Engine End User License Agreement (EULA)

**关键条款摘要：**

| 条款 | 内容 | DynastyVR 影响 |
|------|------|----------------|
| **免费使用** | 产品收入 < $1M USD 时免版税 | ✅ 开发/测试阶段无费用 |
| **5% 版税** | 超过 $1M USD 的毛收入须缴纳 5% | ⚠️ 需财务规划 |
| **收入计算** | 基于毛收入（Gross Revenue），非利润 | 所有收入来源需计入：销售、IAP、订阅、广告、虚拟货币 |
| **季度申报** | 收入超过 $10K USD 后须按季度向 Epic 报告 | 需建立财务合规流程 |
| **Epic 在线服务** | 使用 EOS（在线服务、反作弊等）可获部分版税减免 | 考虑集成 EOS 降低净版税率 |
| **引擎标识** | 须在游戏启动画面保留 "Powered by Unreal Engine" 标识 | UI 设计需预留 |

### 3.2 收入分成详解

**5% 版税计算示例（假设 DynastyVR）：**

```
季度毛收入: $500,000
EOS 抵扣: - $100,000 (假设 EOS 服务抵扣额度)
应税基数: $400,000
版税: $400,000 × 5% = $20,000
```

**收入来源归类：**

| 收入类型 | 是否计入 UE5 版税 | 说明 |
|----------|-------------------|------|
| 游戏买断制销售 | ✅ 是 | 全额计入 |
| 虚拟货币 IAP | ✅ 是 | 购买时计入 |
| 地块销售 | ✅ 是 | 一次性销售收入 |
| 订阅/会员 | ✅ 是 | 按实际收款计入 |
| 虚拟商品（车辆皮肤等） | ✅ 是 | 全额计入 |
| 广告收入 | ✅ 是 | 展示广告收入 |
| 实物商品 | ❌ 否 | 实体商品不在 UE5 EULA 范围 |
| 第三方平台分成 | ⚠️ 视情况 | Steam/Epic Store 已扣的分成前的毛收入计入 |

**风险等级：** 🔴 高 — 收入规模大时版税显著，须提前规划

### 3.3 发布平台限制

**允许的发布平台：** 所有主流平台均支持，包括：
- PC (Steam, Epic Games Store, 自有分发)
- 主机 (PlayStation, Xbox, Nintendo Switch)
- 移动端 (iOS, Android)
- VR 平台 (Meta Quest, SteamVR, PSVR2)

**平台特定注意事项：**

| 平台 | 附加要求 |
|------|----------|
| **Meta Quest** | 须通过 Meta App Lab 或 Store 审核；SideQuest 可用于测试 |
| **Steam** | 须遵守 Steamworks 开发者协议；VR 专区需额外测试 |
| **iOS** | App Store 审核严格（虚拟货币政策）；须遵守 Apple 30% 抽成 |
| **Android** | Google Play 政策较宽松但仍需合规 |
| **China (国内)** | 版号要求（见下文） |

### 3.4 中国大陆特别合规

**游戏版号（ISBN）：**
- 在中国大陆运营需获得国家新闻出版署发放的 **网络游戏版号**
- 版号申请周期：6-18 个月
- 需通过审批后方可正式上线收费
- 可以"体验版"或"测试版"形式先行上线（有时间和人数限制）

**实名认证：**
- 须接入国家新闻出版署的实名认证系统
- 未成年人保护：防沉迷系统（限制游戏时间和消费）

**风险等级：** 🔴 高 — 国内运营合规成本高、周期长

## 4. 用户隐私

### 4.1 GDPR 合规（欧盟用户）

**适用场景：** DynastyVR 面向全球用户，包含欧盟用户时须完全遵守 GDPR。

**核心要求：**

| 要求 | 实施措施 |
|------|----------|
| **合法基础** | 用户同意（精准营销）/ 合同必要性（服务提供）/ 合法利益（安全/反欺诈） |
| **数据最小化** | 仅收集服务必要的用户数据 |
| **知情同意** | 首次登录时弹出隐私政策 + Cookie 同意横幅 |
| **同意可撤回** | 用户可随时在设置中撤回非必要数据收集的同意 |
| **数据可携带** | 用户可导出自己的数据（JSON/CSV） |
| **被遗忘权** | 用户可请求删除账户及所有关联数据 |
| **数据保护官 (DPO)** | 收入超过 €10M 或处理大量个人数据时须指定 DPO |
| **DPIA** | 高风险数据处理须进行数据保护影响评估 |
| **72 小时通报** | 数据泄露须在 72 小时内通知监管机构和受影响用户 |

**DynastyVR 需收集的个人数据及法律依据：**

| 数据类型 | 用途 | 法律依据 | 必要性 |
|----------|------|----------|--------|
| 账号信息（邮箱/用户名） | 身份认证 | 合同必要性 | 必要 |
| 支付信息 | 交易处理 | 合同必要性 | 必要 |
| IP 地址 | 地理定位/反欺诈 | 合法利益 | 必要 |
| 设备信息 | 兼容性/性能优化 | 合法利益 | 必要 |
| VR 使用数据 | UX 改进 | 同意（可选） | 可选 |
| 语音数据 | 语音聊天 | 同意 | 可选 |
| 位置数据 | 自驾路线推荐 | 同意 | 可选 |
| UGC（建造内容） | 社区展示 | 同意 + 合同必要性 | 可选 |

### 4.2 个人信息保护法 (PIPL — 中国)

**核心要求：**

| 要求 | 措施 |
|------|------|
| **单独同意** | 敏感个人信息（生物识别、位置、金融）须获取单独同意 |
| **告知义务** | 隐私政策须以显著方式、清晰易懂的中文呈现 |
| **本地化存储** | 在中国境内收集的个人信息须存储在中国境内服务器 |
| **数据出境** | 向境外传输须通过安全评估或标准合同 |
| **最小必要** | 不得过度收集个人信息 |
| **未成年人保护** | 14 岁以下须获监护人同意 |

**VR 特殊数据类型：**

| 数据 | 敏感程度 | 处理要求 |
|------|----------|----------|
| **眼动追踪数据** | 高敏感 — 可推断健康/心理状态 | 须单独同意，严格加密 |
| **手势/体态数据** | 中敏感 | 须告知用途 |
| **空间映射数据** | 中敏感 — 可识别用户真实环境 | 须告知，不得与第三方共享 |
| **语音生物特征** | 高敏感 | 须单独同意，本地处理优先 |

### 4.3 数据收集告知与同意机制

**实施计划：**

```
用户首次启动 DynastyVR
    │
    ├─ 1. 启动画面："Powered by Unreal Engine" + 隐私政策链接
    │
    ├─ 2. 账号注册页：
    │   ├─ 必要数据收集告知（隐私政策摘要）
    │   ├─ 用户协议勾选
    │   └─ 年龄验证（18+ / 14-18 / <14 需监护人同意）
    │
    ├─ 3. Cookie / 追踪同意横幅：
    │   ├─ 必要 Cookie（默认启用，不可关闭）
    │   ├─ 分析 Cookie（默认关闭，用户可开启）
    │   └─ 广告 Cookie（默认关闭，用户可开启）
    │
    ├─ 4. 首次进入 VR：
    │   ├─ 空间映射告知："设备将扫描您的房间"
    │   └─ 语音功能告知："是否启用语音聊天"
    │
    └─ 5. 设置页随时可调整：
        ├─ 隐私设置面板
        ├─ 数据导出功能
        └─ 账户删除功能
```

**风险等级：** 🔴 高 — VR 设备特殊数据类型的合规边界仍在法律发展中

| 虚拟财产类型 | 用户权利 | 平台保留权利 |

**关键条款：**

1. **用户不拥有虚拟物品** — 用户获得的是**有限的、可撤销的使用许可**

4. **不可转让性** — 虚拟财产不可在 DynastyVR 平台外交易或兑换
5. **继承问题** — 目前法律空白，建议增加"账户继承申请"流程作为备选方案

## 6. 虚拟交易

### 6.1 虚拟货币

**Dynasty Coin 设计原则：**

| 特征 | 设计 | 法律依据 |
|------|------|----------|
| **名称** | 不称"货币"，称"Dynasty 点数/代币" | 避免涉及货币/金融监管 |
| **兑换** | 仅支持人民币/美元 → Dynasty Coin（单向） | 不得提供双向兑换 |
| **获取途径** | 购买 / 游戏内成就奖励 | 双轨制降低监管风险 |
| **有效期** | 建议不设有效期（否则可能被视为预付卡） | 《单用途商业预付卡管理办法》 |
| **未成年人限额** | 月消费上限（参考网易/腾讯标准） | 未成年人保护 |

**法律风险识别：**

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| **虚拟货币 = 金融产品？** | 中国监管禁止虚拟货币交易 | Dynasty Coin 仅单向兑换，不开放交易市场 |
| **洗钱风险** | 虚拟物品可能被用于洗钱 | 大额充值需实名认证 + 异常交易监控 |
| **博彩嫌疑** | 随机抽取（抽卡/开箱）可能被认定为赌博 | 公布概率，不提供变现途径 |
| **预付卡监管** | 大额充值可能触发预付卡管理规定 | 单笔充值上限 ¥500 |

### 6.2 地块交易

**地块交易规则：**

1. **一级市场：** 平台直接出售（新地块）
2. **二级市场：** 用户间转让（平台内交易，平台收取手续费）
3. **禁止三级市场：** 不允许在 DynastyVR 平台外交易地块

**法律注意事项：**

- 地块本质是**虚拟空间使用权**，不是不动产权
- 避免使用"买卖""产权""地契"等不动产术语
- 使用"地块使用权""虚拟空间授权"等表述
- 二级市场交易须全额纳税（平台代扣代缴）

### 6.3 退款政策

**退款框架：**

| 场景 | 退款政策 | 法律依据 |

### 8.3 合规检查流程

**新依赖引入检查清单：**

```
引入新开源依赖前：
  □ 1. 检查许可证类型（SPDX identifier）
  □ 2. 确认非 GPL/AGPL（除非明确允许）
  □ 3. 检查传染性（Copyleft vs Permissive）
  □ 4. 确认有明确的许可证文件（LICENSE / COPYING）
  □ 5. 检查是否有专利相关条款（Apache 2.0 有专利授权）
  □ 6. 检查是否已过期/被替换（安全审计）
  □ 7. 记录在依赖清单中（本文件 + 内部管理表）
  □ 8. 如为 LGPL，确认动态链接方式
  □ 9. 如为 ODbL，确认数据使用方式合规
  □ 10. 法务审核（仅 GPL/AGPL 或高风险依赖）
```

**年度合规审计：**

- 每季度更新依赖清单
- 每年一次全面开源合规审计
