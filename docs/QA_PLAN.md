## 1. 测试策略总览

### 1.1 测试金字塔

```
                    ╱╲
                   ╱  ╲
                  ╱ VR ╲            ← VR 专项测试（眩晕/追踪/空间定位）
                 ╱ 专项  ╲
                ╱────────╲
               ╱   E2E    ╲        ← 端到端场景测试（完整飞行/自驾/建造流程）
              ╱  场景测试   ╲
             ╱──────────────╲
            ╱    集成测试     ╲      ← 系统间交互（Cesium+Physics+Network）
           ╱                  ╲
          ╱────────────────────╲
         ╱      单元测试        ╲    ← 纯逻辑（路线算法/物理公式/数据解析）
        ╱                        ╲
       ╱──────────────────────────╲
      ╱       性能 & 基准测试       ╲  ← 全生命周期贯穿
     ╱                              ╲
    ╱────────────────────────────────╲
```

| 层级 | 覆盖范围 | 目标覆盖率 | 自动化比例 |
|------|----------|-----------|-----------|
| **单元测试** | 路线算法、物理公式、数据解析、工具函数 | ≥ 85% | 100% 自动 |
| **集成测试** | Cesium+渲染、Physics+Vehicles、Network+Sync | ≥ 70% | 90% 自动 / 10% 手动 |
| **E2E 场景** | 完整用户旅程（飞行全程/自驾全程/建造全流程） | 核心路径 100% | 60% 自动 / 40% 手动 |
| **性能测试** | 帧率、内存、带宽、加载时间 | 全平台全场景 | 95% 自动 / 5% 手动 |
| **VR 专项** | 眩晕、追踪精度、空间感知、舒适度 | 关键指标 100% | 30% 自动 / 70% 手动 |

### 1.2 自动化 vs 人工测试比例

| 测试类型 | 自动化 | 人工 | 说明 |
|----------|--------|------|------|
| 单元测试 | 100% | — | UE5 Automation Tool + GoogleTest |
| 集成测试 | 90% | 10% | 场景快照回归 + 人工抽查 |
| E2E 测试 | 60% | 40% | 关键路径自动化，探索性测试人工 |
| 性能基准 | 95% | 5% | Stat UNIT/Stat GPU 自动采集，人工分析 |
| VR 体验 | 30% | 70% | 舒适度主观评估必须人工，追踪精度可自动化 |
| 兼容性 | 80% | 20% | 设备矩阵自动跑，异常人工复现 |

**总体目标：自动化覆盖率 70% 以上，人工聚焦于体验与探索性测试。**

## 3. 性能基准测试

### 3.1 各平台帧率目标

| 平台 | 目标帧率 | 最低可接受帧率 | 测试分辨率 | 测试场景 |
|------|---------|--------------|-----------|---------|
| **PC 高端**（RTX 4090） | 90 FPS | 60 FPS | 4K / DLSS Quality | 全场景 |
| **PC 中端**（RTX 4070） | 60 FPS | 45 FPS | 2K / DLSS Balanced | 全场景 |
| **Quest 3** | 72 FPS | 60 FPS | 原生分辨率 | 优化场景 |
| **Quest 2** | 72 FPS | 50 FPS | 原生分辨率 | 简化场景 |
| **Vision Pro** | 90 FPS | 72 FPS | 原生分辨率 | 全场景 |

#### 测试方法

1. **Stat UNIT 自动化采集**：在每个关键场景放置性能采集点，自动记录 Frame Time、Game Thread、Render Thread、GPU Time
2. **Stat GPU 详细分析**：瓶颈场景使用 `stat gpu` 逐 Pass 分析耗时
3. **帧率曲线记录**：完整录制 300s 场景回放，生成帧率分布直方图
4. **1% Low / 0.1% Low**：关注帧率最低 1% 和 0.1% 的表现，评估卡顿

### 3.2 内存泄漏检测

| 检测项 | 工具 | 通过标准 | 检测频率 |
|--------|------|---------|---------|
| UE5 内存分配追踪 | `memreport -full` | 重复操作 100 次后内存增长 < 5% | 每个 Sprint |
| 纹理内存 | Rendering Audit | 无未释放的 StreamingTexture | 每日构建 |
| World Partition 加载卸载 | 自定义统计 | 切换 20 个区块后内存回到基线 ± 10% | 每日构建 |
| 资源引用泄漏 | Smart Pointer Audit | 无孤立的 UObject 引用 | 每个 PR |
| Cesium Tile 内存 | Cesium Profiler | Tile 卸载后内存释放 > 95% | 每日构建 |

#### 内存泄漏检测流程

```
1. 启动游戏，等待加载完成 → 记录基线内存 (Baseline)
2. 执行测试场景（飞行全路线 / 自驾全路线 / 建造 100 次放置删除）
3. 返回主菜单 → 记录内存
4. 重复步骤 2-3 十次
5. 对比最终内存与 Baseline
6. 差值 < 5% 通过，否则标记泄漏并提交 Bug
```

### 3.3 流式加载性能测试

| 测试场景 | 测试方法 | 通过标准 |
|---------|---------|---------|
| 快速飞越城市 | 以 800km/h 飞越上海→东京，记录加载 | 无白模弹出，纹理渐变过渡 < 1s |
| 从太空到地面 | 从 400km 高空俯冲到地面 | LOD 层级切换平滑，无明显 pop-in |
| 场景切换 | 在纽约和上海间快速切换 | 场景切换时间 < 5s（含纹理流送） |
| 高密度区域 | 飞越曼哈顿中城 | Nanite 代理保持稳定，无崩溃 |
| 多人同时加载 | 10 人同时进入高密度区域 | 每人加载时间 < 8s，服务器无超时 |

#### World Partition 分区加载指标

- **区块加载时间**：< 500ms（低密度区域）/ < 2s（高密度城市区域）
- **区块卸载延迟**：玩家离开区块后 10s 内释放
- **并发加载上限**：同时加载区块数 ≤ 9，防止 I/O 瓶颈

### 3.4 网络延迟压力测试

| 测试项 | 工具 | 条件 | 通过标准 |
|--------|------|------|---------|
| 基础同步延迟 | 自定义 Ping 测量 | 本地局域网 | 位置同步延迟 < 50ms |
| 弱网同步 | Clumsy / Network Link Conditioner | 延迟 200ms + 5% 丢包 | 位置同步延迟 < 400ms，无明显跳变 |
| 高延迟同步 | 同上 | 延迟 500ms + 10% 丢包 | 客户端预测正确，回滚平滑 |
| 大规模同步 | 服务器压力脚本 | 50 人同屏 | 每人位置更新频率 ≥ 10Hz |
| 语音质量 | WebRTC 诊断 | 延迟 300ms + 2% 丢包 | 语音 MOS ≥ 3.5 |
| 断线重连 | 手动断网 30s 后恢复 | 模拟网络切换 | 5s 内自动重连，状态恢复正确 |

#### 网络压力测试场景矩阵

| 场景 | 玩家数 | 网络条件 | 关键指标 |
|------|--------|---------|---------|
| 婚礼活动 | 50 人同屏 | 良好 (延迟 < 100ms) | 帧率 ≥ 45 FPS，同步延迟 < 300ms |
| 自驾车队 | 5 人 | 中等 (延迟 200ms) | 车辆位置偏差 < 2m |
| 全球漫游 | 20 人分散各地 | 良好 | 跨区域切换无缝 |

### 4.1 眩晕度评估标准

#### 评估方法

采用 **Simulator Sickness Questionnaire (SSQ)** 标准量表：

| 维度 | 症状示例 | 评分范围 |
|------|---------|---------|
| **晕动症 (Nausea)** | 胃部不适、出汗、唾液增多 | 0-3 (无/轻微/中等/严重) |
| **眼疲劳 (Oculomotor)** | 眼疲劳、头痛、视物模糊 | 0-3 |
| **方向感障碍 (Disorientation)** | 头晕、定向障碍 | 0-3 |

#### 舒适度标准

| 评级 | SSQ 总分 | 通过标准 | 对应场景要求 |
|------|---------|---------|-------------|
| 🟢 优秀 | 0-5 | 可长时间游玩 (2h+) | 自由漫游、建造 |
| 🟡 良好 | 6-15 | 可中等时长游玩 (1h) | 飞行巡航、自驾平路 |
| 🟠 不佳 | 16-25 | 仅限短时体验 (30min) | 需优化：减少强制运动 |
| 🔴 不可接受 | >25 | 产生强烈不适 | 必须修复后重新测试 |

## 6. Bug 管理与发布流程

### 6.1 Bug 分级标准

| 级别 | 名称 | 定义 | 修复时限 | 示例 |
|------|------|------|---------|------|
| **P0** | 🔴 致命 (Blocker) | 崩溃/数据丢失/核心功能完全不可用 | **立即修复**（阻塞发布） | 启动崩溃、存档损坏、无法进入 VR |
| **P1** | 🟠 严重 (Critical) | 核心功能严重受损，无有效绕过方案 | **当前 Sprint 修复** | 飞机无法起飞、地图加载失败、多人不同步 |
| **P2** | 🟡 重要 (Major) | 功能异常但有临时方案，或次要功能不可用 | **下个 Sprint 修复** | 语音偶尔断连、搜索结果不准确、UI 布局错位 |
| **P3** | 🟢 一般 (Minor) | 体验问题、视觉瑕疵、边缘场景异常 | **Backlog 择机修复** | 文字错别字、动画轻微穿模、非关键提示缺失 |

#### 升级规则

- P3 Bug 累计超过 **50 个** 未修复 → 升级为 P2 评审
- P2 Bug 超过 **2 个 Sprint** 未修复 → 升级为 P1
- 任何 P0 Bug 出现在 Release 候选版本 → 触发热修复流程
